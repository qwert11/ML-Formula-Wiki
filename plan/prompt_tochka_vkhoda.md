# Промпт точка входа

Ты работаешь над проектом

## Твоя роль
Ты исполнитель, который должен последовательно читать проектную документацию и реализовывать систему без самодеятельности.

## Главные правила
1. В первую очередь ориентируйся на документы из папки `plan/tz/`, `plan/tz/modules`, прототипы - `plan/tz/prototype` в порядке возрастания номера файлов: `00_...`, `01_...`, `02_...` и так далее и `specs`.
2. После этого учитывай `plan/tz/trebovaniya_zakazchika/opisanie.txt` как исходное описание заказа.
3. Обязательно читай `plan/tz/vremennye_resheniya.md`. Там перечислены временные допущения по неуточненным требованиям.
4. Если находишь противоречие между макетом, проектными файлами и требованиями заказчика, не придумывай решение молча. Сразу фиксируй вопрос в `plan/faq.md`.
5. Если требование неясно, но по нему есть временное решение, работай по временному решению и отмечай это в `plan/faq.md`.
6. Если требование неясно и временного решения нет, остановись по этому месту и сформулируй конкретный вопрос.
7. Не меняй стек и архитектурные ограничения без явного вопроса в `plan/faq.md`.
8. Двигайся по дорожной карте `roadmap.md` - roadmap — это только порядок выполнения задач, он не содержит требований и не влияет на логику реализации.
10. questions_for_client.md - файл требований к заказчику не содержит требований и не влияет на логику реализации.
11. На файлы в папке "plan\audit" не обращай внимания. Это промежуточный аудит который статичен на определенный срез.
12. План может менятся и дополняться - для этого могут быть созданы Файлы версии плана

## Почему именно так
- Нужно быстро собрать аккуратное демо.
- Нужно не подвисать на слабом железе.
- Нужно легко заменить моки на backend.
- Нельзя раздувать проект тяжелыми зависимостями без пользы.

## Приоритет источников
Приоритет принятия решений такой:
1. Файлы из `plan/tz`
2. Временные решения
3. Исходное описание заказчика

Если источники конфликтуют, сообщай об этом через `plan/faq.md`.

## Как работать с документацией
Перед началом:
1. Прочитай `plan/tz/00_project_summary.md`.
2. Затем прочитай остальные numbered-файлы по порядку.
3. Затем прочитай спецификации.
4. Затем проверь `plan/tz/vremennye_resheniya.md`.
5. Затем приступай к выполнению.

## Как фиксировать прогресс
- В `plan/roadmap.md` отмечай выполненные этапы.
- В `plan/faq.md` записывай:
  - вопросы к требованиям
  - найденные противоречия
  - что сделано дополнительно
  - что не сделано
  - какие временные решения были использованы
